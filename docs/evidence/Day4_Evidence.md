# Day4 Evidence - 发布表单与数据新增

## 1. 今日完成内容

今天完成了四类业务的**发布表单**页面开发、**表单校验**逻辑和**数据新增**功能。具体包括：在四个 API 模块中新增 `createXxx()` POST 方法；创建 `FormField.vue` 通用表单字段组件；重构 `PublishView.vue` 实现统一的发布页面，支持切换四种业务类型并显示对应的专属字段；实现 `clearErrors` + `validateForm` 校验函数，对必填项和数值范围进行基础校验；提交成功后调用对应 API 写入 JSON Server 并跳转到列表页。

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 需要展示商品基本信息和交易条件，价格和成色直接影响买家决策 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领两种类型，并描述物品特征和丢失时间 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 需要说明拼单类型、目标人数和截止时间，让用户判断能否参与 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 需要说明任务内容、取送地点和酬劳，是跑腿业务最核心的信息 |

四个业务共享标题、地点、描述三个通用字段。标题让用户快速了解内容；地点让用户判断是否在自己附近；描述字段用于填写详细的补充说明。每个业务在此基础上增加自己独有的字段，例如二手交易需要价格和成色，拼单需要人数目标。

## 3. 表单校验规则

- **标题**：不能为空。标题是每条信息的第一印象，必须填写。
- **地点**：不能为空。校园交易依赖位置信息，没有地点用户无法判断距离。
- **描述**：不能为空。描述是信息的正文部分，不能省略。
- **二手交易价格**：必须大于 0。价格为 0 或负数没有实际意义。
- **二手交易商品分类**：不能为空。需要知道闲置物品属于哪一类。
- **二手交易成色**：不能为空。成色影响交易是否达成。
- **失物招领物品名称**：不能为空。必须知道丢失或拾到的是什么物品。
- **失物招领发生时间**：不能为空。时间信息有助于判断是否匹配。
- **拼单类型**：不能为空。需要知道拼的是餐、资料还是其他。
- **拼单目标人数**：不能少于 2 人。单人不能称为拼单。
- **拼单/跑腿截止时间**：不能为空。超时信息没有意义。
- **跑腿任务类型**：不能为空。需要知道是取快递还是代买。
- **跑腿酬劳**：不能为负数。酬劳不能是倒贴钱。
- **跑腿取件/送达地点**：不能为空。起终点缺一不可。

所有校验规则都围绕一个原则：缺失该字段后用户无法正常使用这条信息。不引入额外规则，保持校验逻辑简单易懂。

## 4. AI 协作记录

使用的 AI 工具为 Claude（opencode 内置模型），在 VS Code 终端中以对话方式完成。

核心提示词方向：要求 AI 在已有的 Vue3 + Vite + TypeScript + Axios + JSON Server 项目中实现 PublishView.vue 发布表单，支持四类业务，包含表单校验和 POST 提交。

AI 生成的内容包括：
- 在 trade.ts、lostFound.ts、groupBuy.ts、errand.ts 中添加了 `createXxx()` POST 方法
- 创建了 FormField.vue 表单字段组件
- 生成了 PublishView.vue 的完整表单模板，包含四种业务的字段
- 实现了 validate() 校验函数和 handleSubmit() 提交逻辑
- 提供了重置表单和成功/失败提示

AI 生成内容中发现的不合理之处：
- 类型专属字段使用 `v-if` 散插在通用字段之间，导致表单顺序不统一（例如价格字段出现在地点之前）。调整为 `<template v-if>` 分组放在描述字段之后。
- 使用 `buildPayload()` 中间函数组包后再调用 API，增加了理解成本。改为每个类型单独构造对象并直接调用 `createXxx()`。
- 校验函数 `validate()` 使用单行缩写风格，对教学实训不友好。改为带花括号的 `validateForm()` 完整写法。
- 提交成功后统一跳转到首页 `/`，不合理。改为跳转到对应业务列表页。

## 5. 人工调整内容

- 将散插的 `v-if` 字段重组为四个 `<template v-if="publishType === 'xxx'">` 分组
- 将 `buildPayload()` 中间层去掉，改为每个类型独立传参，字段映射更直观
- 将 `validate()` 单行缩写风格改为带花括号的 `validateForm()` + `clearErrors()` 写法
- 将 `getCurrentTime()` 改为 `yyyy-MM-dd HH:mm` 本地格式化，符合中文习惯
- 路由跳转从 `/` 改为 `/trade`、`/lost-found`、`/group-buy`、`/errand`，对齐 router 配置
- 修改表单控件样式从 `display: flex` 为 `display: grid`，布局更规整
- 添加 `input`、`select`、`textarea` 的 `width: 100%` 和 `box-sizing: border-box`，避免溢出

## 7. 遇到的问题与解决方法

**问题**：表单中的类型专属字段使用 `v-if` 散插在通用字段之间，例如价格字段放在了标题之后、地点之前。当用户切换发布类型时，字段出现和消失的位置不统一，体验不好。另外，切换类型后之前填写的字段值会残留，用户手动清空比较麻烦。

**解决方法**：将所有类型专属字段用 `<template v-if>` 统一放在描述字段之后、提交按钮之前。通用字段（标题、地点、描述）始终固定在表单上部，专属字段集中在表单中部，按钮固定在底部。同时提供了 `resetForm()` 函数，用户点击重置按钮或切换类型时调用（Day4 阶段切换类型不强制重置，但函数已写好备用）。

**另一个问题**：写入文件时使用了中文路径名"桌面"而非英文"Desktop"，导致文件被写到了错误的位置，Git 仓库没有跟踪到这些文件。通过 `cp` 命令将文件从中文路径复制到英文路径解决了问题。

## 8. 今日反思

**发布表单**是用户与系统交互最直接的方式——用户通过表单把信息提交到平台，其他用户才能看到和响应。没有表单，页面只有静态数据，应用就没有"写入"能力。**表单校验**是在用户提交前把关，防止不完整或错误的数据进入数据库，既提升了数据质量，也避免了后端处理脏数据的问题。**数据新增**（POST 请求）让应用从只读变为可写，是 Web 应用从展示型转向交互型的关键一步。四类业务共享同一个发布页面，通过 `publishType` 字段控制显示内容，代码复用率高，后期新增业务类型只需添加新的 `<template>` 分组和 API 调用，扩展成本很低。整个表单开发过程中，组件化（FormField）和模块化（API 模块）让代码结构清晰，适合教学实训场景继续迭代。
