## 2. 表单字段设计

### 二手交易 (trade)
| 字段 | 必要性 |
|------|--------|
| 商品分类 | 区分闲置品类，方便买家浏览 |
| 价格 | 二手交易核心信息，必须明确标价 |
| 成色 | 影响买家购买决策的关键信息 |

### 失物招领 (lostFound)
| 字段 | 必要性 |
|------|--------|
| 信息类型 | 区分"寻物"和"招领"，避免混淆 |
| 物品名称 | 明确丢失/拾到的物品是什么 |
| 发生时间 | 帮助失主判断是否为自己遗失的物品 |

### 拼单搭子 (groupBuy)
| 字段 | 必要性 |
|------|--------|
| 拼单类型 | 区分拼餐、资料团购、运动等 |
| 目标人数 | 拼单的核心——需要多少人才能成团 |
| 截止时间 | 让参与者知道何时截止 |

### 跑腿委托 (errand)
| 字段 | 必要性 |
|------|--------|
| 任务类型 | 区分取快递、代买、搬运等 |
| 酬劳 | 跑腿任务的核心报酬信息 |
| 取件地点 | 从哪取 |
| 送达地点 | 送到哪 |
| 截止时间 | 任务需要在什么时间前完成 |

## 3. 校验规则设计

| 字段 | 校验规则 | 原因 |
|------|---------|------|
| title | 必填 | 每类业务都需要标题描述 |
| location | 必填 | 位置信息对校园交易至关重要 |
| description | 必填 | 详情描述必不可少 |
| category (trade) | 必填 | 需要知道商品分类 |
| price (trade) | > 0 | 价格必须为正数 |
| condition (trade) | 必选 | 成色影响交易判断 |
| itemName (lostFound) | 必填 | 必须知道丢失/拾到的物品 |
| eventTime (lostFound) | 必填 | 时间有助于定位失物 |
| groupType (groupBuy) | 必填 | 需要知道拼单类型 |
| targetCount (groupBuy) | ≥ 2 | 拼单至少需要 2 人 |
| deadline (groupBuy/errand) | 必填 | 拼单和跑腿必须有截止时间 |
| taskType (errand) | 必填 | 需要知道任务类型 |
| reward (errand) | ≥ 0 | 酬劳不能为负数 |
| from (errand) | 必填 | 取件地点必须明确 |
| to (errand) | 必填 | 送达地点必须明确 |

## 5. 人工调整内容

- 将散插在通用字段间的类型专属字段重组为 `<template v-if>` 分组，放在描述字段之后，结构更清晰
- 将 `buildPayload()` 中间函数改为每个类型独立构造对象，字段映射更直观
- 将 `validate()` 单行缩写风格改为带花括号的完整写法，更易阅读和理解
- 将 `getCurrentTime()` 改为本地格式化（`yyyy-MM-dd HH:mm`），而非 ISO 格式，更符合中文阅读习惯
- 路由跳转路径对齐项目中 router 配置（`/trade`、`/lost-found`、`/group-buy`、`/errand`）

## 6. 遇到的问题与解决方法

**问题**：表单中的类型专属字段散插在通用字段之间，导致表单顺序混乱（价格在标题之后但在地点之前）。

**解决方法**：使用 `<template v-if="publishType === 'xxx'>"` 将每个类型的专属字段统一分组，放在描述字段之后、提交按钮之前，无论切换哪种类型，通用字段和按钮位置始终保持一致。

## 7. 今日反思

发布表单是用户与系统交互的核心入口。通过 FormField 组件封装字段展示逻辑（标签、必填标记、错误提示），避免了每个字段重复编写模板代码。四类业务共享同一个表单页面，通过 `publishType` 动态切换字段，保持了代码的 DRY 原则。表单校验以基础必填和数值范围为主，不引入第三方校验库，降低了学习成本。POST 请求直接使用已有 API 模块，请求逻辑与页面逻辑保持分离。
